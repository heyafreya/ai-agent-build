Project Summary

Welcome

This page hosts my entry for the Boston Kubernetes Meetup AI Agent Build Competition — a skills-building challenge where the goal is to practice a new skill, document what you learned, and publish it so others can learn too.

I hadn't built an agent before and don't have extensive Kubernetes experience. This project let me learn both at the same time: an AI agent that monitors a Kubernetes cluster and explains what's wrong in plain English.

How it works

The agent uses a ReAct loop — the LLM decides which tools to call, investigates failures, and iterates until it has enough evidence to explain what's wrong. A deterministic severity scorer runs alongside as a safety net.

User request -> agent.py (ReAct loop) -> tools: get_pods, get_logs, describe
                                    -> (parallel) alerts.py (hard rules)
                                    -> combined output + trace data

Tools the agent can call

ToolWhat it does
GET_PODSList all pods with status, restarts, readiness
GET_LOGSRetrieve recent log lines from a pod
DESCRIBE_PODGet detailed events and conditions

What I learned

LLMs fabricate data without guardrails. The model invents pod names that don't exist unless you give it an explicit list. Adding EXISTING PODS to the prompt fixed this.

Deterministic scoring is a safety net. LLMs aren't deterministic — the same input can produce different outputs. Pure-function severity rules (alerts.py) run alongside the LLM so the severity block is always correct.

Regex parsing is fragile. The original agent parsed tool calls with regex. Any markdown formatting broke it. Structured JSON output made it reliable.

Traces make agents transparent. Capturing every step of the ReAct loop turns a black box into something auditable. Click "Agent Trace" after running an analysis to see it.
